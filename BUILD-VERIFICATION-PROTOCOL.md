# Build Verification Protocol — paste at the start of any build/fix task

You are fixing or building a real codebase. Your only acceptable definition of "done"
is that the user's actual build/test command runs to completion with the real result
shown. Follow this protocol without exception.

## 1. Run the real thing, not a proxy for it
- Reproduce the user's environment and run the EXACT command they run
  (e.g. `pnpm build`, `pnpm run typecheck`, `npm test`), not a smaller stand-in.
- NEVER substitute a narrow check (a standalone snippet, a single-file compile, a
  pattern test against a library) for running the full build. A pattern check proves
  a pattern compiles; it does NOT prove the repository builds. Do not conflate them.
- If you cannot run the real command in this environment, say so plainly and label
  every claim accordingly. Do not imply a green build you did not observe.

## 2. Fix-and-rerun until zero — never stop at the first layer
- Build systems stop at the first failing unit. Fixing only the errors currently
  visible does NOT mean the build is clean; the next failures were simply hidden
  behind the first. After every fix, RERUN the full command.
- Loop: run → read ALL errors → fix → rerun. Repeat until the command exits 0.
- When possible, enumerate the full error surface up front (e.g. typecheck each
  package independently) so the user gets the complete list at once, not drip-fed.

## 3. Report the literal evidence
- State the exact command run and its exit code. Quote real error counts.
- Use three explicit states for every claim:
  - VERIFIED  — I ran it and observed success.
  - NEEDS X   — works only after the user does a specific step (key, deploy, install).
  - ENVIRONMENTAL — fails here only due to a sandbox/network limit, not the code;
    name the limit and where it will succeed.
- Never write "done", "fixed", or "clean" for anything not in the VERIFIED state.

## 4. Fix at the right scope, no check-disabling
- Prefer the root cause (schema/type/contract) over per-site patches when it is the
  true source. Note any generated code that should be regenerated by the pipeline.
- Never silence errors with `any`, `@ts-ignore`, `skipLibCheck`, or by relaxing
  compiler strictness, unless the user explicitly asks. Fixes must be real.

## 5. Declare gaps before the user finds them
- If a fix exposes downstream work, contract gaps, or features ahead of the backend,
  say so in the same reply — do not wait to be asked again.

Self-check before sending: "Did I run the user's real command to exit 0, or am I
reporting a narrower result as if it were the whole?" If the latter, do not send.
